# flagship use cases (method test set)

these four use cases are the initial test set for validating the shared methodology.

each case was identified using the discovery methods documented in [methodology.md](./methodology.md) and follows the full adoption-eval flow: discover → map → evaluate → plan.

## included scenarios
1. `01-internal-knowledge-alignment.md` — operational, internal teams
2. `02-revops-pipeline.md` — revenue, CRO buyer
3. `03-creative-content-versioning-pipeline.md` — creative, marketing buyer
4. `04-cfo-financial-intelligence.md` — finance, CFO buyer

## why this set
- **use case 01** tests knowledge access, alignment, and permission-aware retrieval
- **use case 02** tests lead enrichment, CRM intelligence, tool consolidation, and pipeline hygiene
- **use case 03** tests creative generation, structured adaptation, and versioned multi-channel delivery
- **use case 04** tests hard financial boundaries, MCP tool-calling architecture, and human approval gates

## coverage
together, they cover four distinct adoption contexts:
- **operational** (01) — internal productivity
- **revenue** (02) — sales and pipeline
- **creative** (03) — content and marketing
- **finance** (04) — reporting and governance

four different buyers, four different risk profiles, same methodology.

## what makes each case complete
every case includes:
- **discovery method** — how the workflow was identified as an AI candidate
- **candidate signals** — the evidence that flagged it for evaluation
- **why AI specifically** — why AI is the right lever, not just generic automation
- **could / should / must-stay-human** — the adoption-eval decision framework applied
- **hard boundaries** — non-negotiable constraints enforced by design
- **success metrics** — measurable outcomes for a pilot
- **failure modes** — what to teach users before they start
